# jarvis-team 架构优化清单

> 状态: proposed
> 最后更新：2026-06-16
> 目标：以“高内聚、低耦合、易维护、可扩展到多 IM”为前提，整理下一次开发的优化重点

---

## 一、优化目标

下一次开发不再以“继续堆功能”为主，而是先收敛系统边界，让飞书从“系统本体”降级为“一个 IM 适配器”。

目标原则：

- **高内聚**：同一模块只负责一类变化
- **低耦合**：核心流程不直接依赖飞书、lark-cli、chat_id 等平台细节
- **可替换**：未来可接入 Slack、企业微信、Telegram 等其他 IM
- **可维护**：状态、权限、身份、执行链路都有唯一事实源
- **可演进**：后续增加 Agent、审批、记忆、存储时不需要重写主流程

一句话约束：

> 核心层里不应该出现 `Feishu`、`Lark`、`chat_id`、`verificationToken` 这类平台专有名词。

---

## 二、当前主要不规范点

### P1. 主入口承担过多职责

当前 [src/index.ts](/Users/muliminty/project/jarvis-team/src/index.ts) 同时承担：

- HTTP 接入
- 飞书签名校验
- 事件标准化
- 任务创建
- 步骤创建
- Agent Loop 执行
- 飞书回复
- 轮询引擎
- WebSocket 启动
- Web UI 路由

问题：

- 入口层和应用层混在一起
- 很难做模块替换和单元测试
- 任意接入方式变化都会影响主流程

优化要求：

- `src/index.ts` 只保留启动、装配、路由注册
- 业务流程迁移到独立的 use case / service

### P1. 业务核心被飞书平台字段污染

当前编排和执行流程直接围绕飞书的 `chat_id`、飞书消息结构、飞书签名规则工作。

问题：

- 业务层无法脱离飞书独立存在
- 接入第二个 IM 时只能复制一套分支逻辑
- 状态模型和 Prompt 上下文被平台字段反向污染

优化要求：

- 定义统一的 `InboundEvent`
- 用 `conversationId` 代替 `chat_id`
- 用 `platform` 标识来源，而不是让全链路直接识别飞书

### P1. Tool 层直接耦合 lark-cli

当前 `tool-service` 的核心执行方式是直接调用 `lark-cli`。

问题：

- 工具层不是通用能力层，而是飞书命令封装层
- 无法为不同 IM、不同 Docs、不同审批服务复用
- 测试时难以替换为 fake/mock 实现

优化要求：

- 将“发消息 / 发卡片 / 建文档 / 搜索文档”抽象为端口
- `tool-service` 依赖接口，不直接依赖 `lark-cli`
- 飞书实现下沉到 `adapters/im/feishu` 或 `adapters/feishu`

### P1. 状态服务不是实际的事实源

当前 `state-service` 只在任务创建时参与，执行过程中的轮次、消息、审批、工具调用没有完整回写。

问题：

- 状态不可回放
- 审计链不完整
- 保险丝与任务状态脱节
- 重启后恢复能力弱

优化要求：

- 所有任务状态推进统一通过状态服务
- 记录 task / step / message / tool-call / approval 事件
- 为后续 PostgreSQL 持久化留出清晰的 repository 边界

### P1. Agent 身份模型不统一

当前系统同时存在两套身份来源：

- `agents/*.json` 中的 `feishuBot.appId/appSecret`
- `lark-cli profile` 作为实际执行身份

问题：

- “配置已绑定”不等于“运行可用”
- Agent 身份的真实来源不明确
- 多 Bot 路由难以建立一致性

优化要求：

- 明确唯一身份来源
- 运行时统一从 Agent Profile 解析 outbound 身份
- 禁止一部分逻辑读 JSON，另一部分逻辑依赖本地 CLI profile 猜测身份

### P1. 编排器定义与实际职责不一致

当前 `orchestrator` 自称总指挥，但运行时只做简单意图分类和步骤生成，真正执行仍在入口直接调用 Agent Loop。

问题：

- 模块名和模块行为不一致
- 后续接手的人会误判职责边界
- 多 Agent 协作无法真正闭环

优化要求：

- 编排器负责真正的任务路由和状态推进
- 入口层不能直接跳过编排器去跑执行链

### P1. 步骤依赖模型不规范

当前 `planSteps()` 返回的 `dependsOn` 使用伪造的 `step-0` / `step-1`，但实际创建的 Step ID 是随机生成的。

问题：

- 依赖关系不可解析
- DAG 只是看起来存在，实际上不可执行

优化要求：

- 先定义稳定的步骤引用方式
- 要么使用索引后映射真实 ID
- 要么显式使用可预测的业务 Step Key

### P2. 权限和审批规则来源分散

当前权限判断来自三处：

- 全局 `PermissionConfig`
- Tool 自身 `requiresApproval`
- Agent 配置里的 `approvalRequired`

问题：

- 规则没有统一决策中心
- 行为不容易预测
- 后续接审批流会越来越乱

优化要求：

- 统一为一个 `Policy Engine`
- 输入：Agent、Tool、Action、Resource、Risk
- 输出：`allow / deny / ask`

### P2. 接入方式没有统一抽象

当前 HTTP 回调、WebSocket 长连接、轮询三种方式都直接调用相同处理函数，但没有先汇总为统一事件模型。

问题：

- 接入变化直接影响业务流程
- 重复逻辑分散在不同入口

优化要求：

- 各入口先转换成统一 `InboundEvent`
- 核心逻辑只消费标准化事件

### P2. 配置与常量散落

当前模型名、轮询间隔、CLI 超时、目录常量、默认 chat seed 等分散在多个文件。

问题：

- 运维和开发修改成本高
- 容易出现多处定义不一致

优化要求：

- 建立统一 `config` 模块
- 明确区分运行时配置、开发默认值、环境变量来源

### P2. 存在环境硬编码

当前代码中包含固定私聊 `chat_id` seed 等环境依赖。

问题：

- 不可迁移
- 不可复用
- 不利于测试和开源

优化要求：

- 禁止提交环境专属 ID
- 需要 seed 时通过配置或数据初始化完成

### P3. Web UI 以内联字符串维护

当前 `src/web/index.ts` 直接内联整页 HTML/CSS/JS。

问题：

- 不利于维护
- 不利于前后端边界演进
- 不利于 UI 测试

优化要求：

- 原型阶段可保留
- 若继续演进，拆成静态资源或单独前端模块

## 三、推荐的目标架构

推荐按下面的边界重组：

```text
src/
  core/
    domain/
    ports/
  application/
    use-cases/
    orchestrator/
    agent-runtime/
  adapters/
    im/
      feishu/
      slack/
      wecom/
    persistence/
    approval/
    docs/
  bootstrap/
    http-server.ts
    ws-server.ts
    container.ts
```

### 模块职责

#### `core/domain`

只放核心概念：

- Conversation
- Message
- Task
- TaskStep
- Agent
- Approval

#### `core/ports`

只定义接口，不写平台实现：

- `InboundEventPort`
- `IMOutboundPort`
- `TaskRepository`
- `ApprovalService`
- `MemoryRepository`

#### `application/use-cases`

负责具体业务动作：

- `handleIncomingMessage`
- `createTaskFromMessage`
- `runTask`
- `sendReply`

#### `application/orchestrator`

负责任务编排、步骤推进、Agent 选择、状态推进。

#### `application/agent-runtime`

只负责：

- 组装 prompt
- 选择工具
- 调用 Agent Loop

不直接关心飞书和具体平台消息格式。

#### `adapters/im/feishu`

只负责飞书适配：

- 验签
- 解析飞书消息
- 飞书消息回发
- WebSocket / HTTP / Polling 接入转换

#### `bootstrap`

只负责依赖注入、服务启动、路由挂载。

## 四、下一次开发的优先级清单

### 第一优先级：先拆边界，不加新功能

1. 将 `src/index.ts` 拆成 `bootstrap + route handlers + use cases`
2. 定义统一 `InboundEvent`
3. 定义统一 `IMOutboundPort`
4. 把飞书相关逻辑迁移到 adapter 层
5. 让核心流程不再直接读写 `chat_id`

### 第二优先级：收敛身份、权限、状态

1. 明确 Agent 身份唯一来源
2. 统一权限/审批决策入口
3. 让状态服务接管执行过程
4. 为 PostgreSQL 持久化留 repository 接口

### 第三优先级：整理执行链路

1. 让 orchestrator 真正负责调度
2. 修正 step dependency 模型
3. 记录 tool call 和审批事件
4. 清理环境硬编码

### 第四优先级：再考虑扩展

1. 增加多 IM 接入
2. 增加更多 Agent 角色
3. 增加完整审批流
4. 增加记忆和上下文工程

---

## 五、本轮重构的验收标准

满足以下条件，才能认为架构优化达到第一阶段目标：

- 核心业务流程不直接依赖飞书字段
- 新增一个 IM 时，不需要改 orchestrator / agent-runtime / state-service
- `tool-service` 不直接依赖 `lark-cli`
- `state-service` 能看到任务执行过程，而不是只有创建记录
- `src/index.ts` 不再承担业务编排职责

---

## 六、暂时不建议优先做的事情

在边界没拆干净前，以下事项都不建议优先推进：

- 继续增加 Agent 角色
- 大规模扩工具数量
- 深化飞书卡片审批
- 直接上复杂记忆系统
- 先做 UI 美化

原因：

- 会把当前耦合结构继续固化
- 后续重构成本更高

---

## 七、建议的开发顺序

建议下一次开发严格按以下顺序推进：

1. 抽象统一事件和 outbound port
2. 拆分飞书 adapter
3. 收敛入口层职责
4. 整理状态服务与编排器
5. 统一权限/审批策略
6. 再评估多 IM 和多 Agent 扩展

---

## 八、给下一次开发的提醒

这次优化的重点不是“写更多代码”，而是“减少错误边界和变化扩散面”。

判断一个改动值不值得做，可以只问两个问题：

1. 这个变化是不是只应该影响一个模块？
2. 如果明天换成 Slack，这里还需要改吗？

如果答案分别是“不是”和“需要”，那通常说明边界还没拆对。
